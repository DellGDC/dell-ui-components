Product Stack Elements: Details

Stack Label

The Stack Label is used to highlight a unique product attribute based on logic that incorporates legal requirements, lifecycle needs, timed events, and marketing differentiation.

Code/Logic

  • Optional. Allow ON only for items at SKU/Order Code (OC) level.
  • Automatically toggle OFF for items at Model and Franchise levels.
  • Text only. No special formatting.
  • Use an inheritance model to provide Bugs that can cascade across Franchise, Model, and OC levels.
  • In situations where there is no required legal need, display of the Bug is based on business logic and priority rankings. For example, business logic may specify that timed event Bugs can be overridden by ATS (available-to-sell) conditions, so that Cyber Monday Special text is replaced by Only [N] Left! when inventory levels hit a certain threshold.

UX

  • Not clickable. No interaction.

Visual Notes

  • Text is centered or left-aligned to match with other elements in the stack.
  • Text wraps as necessary.

Content Usage

  • Business-controlled copy assigned at the SKU/OC level. Content is ideally concise to keep the bug on one line, without wrapping.
  • The stack label can be displayed in any Product Stack context, including search results, Browse, product carousels, and Cart.
  • When the stack label is manually maintained for each Order Code, the Business must create a solid content strategy for assigning useful differentiators to each OC. When all OCs on a page show the same bug text (for example, “Dell Recommended”), the stack label no longer serves its purpose as a useful differentiator.
  • DCX recommends that the stack label be suppressed in Product Stacks for items whose Product Titles are also using differentiator terms (such as Good, Better, or Best or Fast, Ultra Fast, or Turbo Charged). This is to allow the user to focus on one key differentiator per item and avoid any potential conflicts between differentiator content in the Bug and the Title.

Asset Substack

Provide users with visual images of the product and/or its options. Use the violator to only message a differentiating feature or offer on the page.

Code/Logic

  • General
  • Allow ON for items at any level, except on pages where the Image Gallery pattern is being used to display product assets.
  • Main Image
  • Can be linked or unlinked. When linked, the Main Image links to the same location as the Product Title (usually the PD page).
  • On PD pages, the Main Image is not linked and has no interaction.
  • Image Violators
  • Optional
  • 1 violator per image

UX

  • General
  • Can be clickable to the same destination.
  • Main Image
  • N/A
  • Image Violators
  • 0-3 max violators per page (exception: search results pages, because multiple results can be pulled without business control)

Visual Notes

  • Image Violators
  • Place at the top of the product on the right side.
  • Text is right aligned and wraps as necessary

Content Usage

  • Main Image – Legal Requirements
  • When the base model price is displayed, the base model image must be displayed.
  • If an image showing an upgrade option is displayed, the price must reflect the upgrade charge.
  • When showing images/assets for a pre-configured Order Code or SKU, use the exact pre-configured OC/SKU images, including color or other visual options.
  • When showing a family image, you can include an upgradable option as long as the base model is also shown.
  • When showing an image for a bundle, indicate all the items included in either the main image or an image violator.
  • When showing images/assets for a configurable Order Code, you can use: (a) The base model images/assets. No Legal Disclosure is required. (b) A non-base model image/asset, but a Legal Disclosure must be attached indicate that the image shown is not the base model.
  • Also see requirements under Pricing Substack for details on image and pricing requirements.Image Violators
  • Image violators provide business controlled content that sits above the main image to call out a key differentiator.
  • 1-5 words max in any language

Product Title

The Product Name, allow for an optional display of product description, and provide a link to more product details (usually the PD page).

Code/Logic

  • General
  • REQUIRED
  • Can be linked (optional). When linked, the Title always links to the same location as the Main Image of the Asset Substack (usually the PD page).
  • On the PD page itself, the Title is not linked and does not have any interaction.
  • On Search Result pages, Title can take OnHover state (optional).
  • Values
  • For Systems Order Code level stacks, use the OC/SKU name (product name).
  • For Systems Model level stacks, use the Model name.
  • For Systems Franchise level stacks, use the Franchise name.

UX

  • N/A

Visual Notes

  • Text wraps as necessary.

Content Usage

  • General
  • SNP titles come from a variety of 3rd party data sources; titles from various data feeds may be inconsistent compared to others. For this reason, truncation can be dangerous because the differentiator in the title may be the very last item.
  • Best Practices for using Title as a Differentiator
  • For Order Code/SKU level stacks, assign a product title that quickly – in the first few words - helps users understand what differentiates one OC from another.
  • For a Bundle or Buddy Order Code, indicate the bundle in the Product Title.

Ratings Substack

To provide customers with at-a-glance ratings for a given product to help customers during decision-making.

Code/Logic

  • General
  • Allow ON for items at any level.
  • Ratings Calculations
  • SKU: Ratings shown for a SKU display the ratings received only for that SKU
  • Order Code: Ratings shown for an OC item are actually the ratings received for the Model. All OCs under a model display the same Ratings substack.
  • Model: Ratings shown for a Model include all the ratings received for the model.
  • Franchise: Ratings a franchise level must be calculated by averaging all ratings from all models within the franchise.
  • Globalization
  • When the Ratings substack is displayed in a Right-to-Left orientation, the stars icons must flip, so that the number of stars increases from right to left, rather than left to right. The number of ratings should also flip to read from right to left.
  • See the Ratings & Reviews pattern for more information.

UX

  • See the Ratings & Reviews pattern for more information.

Visual Notes

  • See the Ratings & Reviews pattern for more information.

Content Usage

  • General
  • The Ratings Substack can be displayed in any Product Stack context (including search results, Browse, or “short stacks” such as product carousels). However, because Ratings are intended for use mostly during the decision-making process, the Ratings Substack is typically not shown in Cart (post-decision state).
  • See the Ratings & Reviews pattern for more information.

Pricing Overview

The pricing overview displays the final price and is only visible in a simplified version of the product stack, typically for small or extra small screen sizes. Upon clicking more details, the full pricing sub stack is then displayed, and only then could the pricing sub stack and overview pricing be displayed together. The pricing sub stack and overview pricing can never be displayed together upon first arrival at the product stack.

Important! The pricing overview is automatically generated through the pricing sub stack. See the Pricing Sub Stack to learn about the legal requirements of the final price.

Code/Logic

  • General
  • Can be displayed in a simplified version of the product stack.
  • The overview pricing and pricing substack can never be displayed at the same time without first clicking “more details”.
  • Final price is pulled from the pricing sub stack.

UX

  • General
  • If the pricing overview is used with the “More Info” control, clicking or tapping “More Info” will expose the full pricing stack.

Visual Notes

  • N/A

Content Usage

  • General
  • Price only is pulled from the pricing sub stack, labels aren’t used.
  • If using the pricing overview, the price should not be used in the title.

Tech Specs - Selected Specs

Provide a visual overview of the specs for a product and how it compares to other products.

Code/Logic

  • General
  • Allow ON for items at the OC level only.
  • Maximum use of four specs. For desktop and laptops, the first spec is always processor and the OS is always the second spec.
  • The ratings scale of each spec will be manually determined by business.

UX

  • Not clickable. No interaction.

Visual Notes

  • Text wraps as necessary.
  • Visual tech specs are accessibility compliant
  • Visual treatment is usually used in horizontal stack layouts such as Search Results.
  • Specs use text only in small/extra small, as the visuals are too hard to compare on small screens

Content Usage

  • Ensure that lists are MDA compliant, mainly by listing processor and OS first and second respectively. See the MDA Compliance CORAL wiki for more information.

Featured Specs

Provide a callout for a featured tech spec, such as touch-enabled or giant hard drive.

Code/Logic

  • General
  • Allow ON for items at any level.
  • Only simple text and image.

UX

  • Not clickable. No interaction.

Visual Notes

  • Text is centered or left-aligned to match with other elements in the stack.
  • Text wraps as necessary.

Content Usage

  • Use the differentiator in conjunction with these other items to fully implement the product content strategy:
    • Product Bug
    • Product Title
    • Tech Specs
    • Product Image, and Image Violators

Product Blurb

Provide a short marketing statement about the product. Note that with the addition of the Tech Specs field, the Blurb is no longer used for showcasing tech specs or product dimensions.

Code/Logic

  • General
  • Allow ON for items at any level.
  • Text only, no special formatting.

UX

  • Not clickable. No interaction.

Visual Notes

  • Text is centered or left-aligned to match with other elements in the stack.
  • Text wraps as necessary.

Content Usage

  • Use the Blurb in conjunction with these other items to fully implement the product content strategy:
    • Product Bug
    • Product Title
    • Tech Specs
    • Product Image
  • Product blurb is owned by the business.

Special Offers

Product Offers always link to further product details. They can be displayed in one of three variants.

Single offer: A single offer launches a modal where the user can read the offer and proceed from an action link.

Multiple offers: Multiple Offers launch a workflow window process where each of the offer links continue to the specific product offer.

Deals offer: Deal offers are always listed as links and can only appear in deals product stacks

Code/Logic

  • Allow ON for items at any level
  • Text only, no special formatting.
  • Offer details displayed differently depending on the number of offers and where the product stack is being displayed.
Show Offer Process Hide Offer Process

UX

  • 1 Special Offer
  • Utilize a modal
  • 2+ Special Offers
  • Utilize a workflow
  • Product Stacks on Deals Page Pages
  • Special Offers are listed out. They must be clearly distinguishable from the next.
  • Utilize a modal if additional details for each offer are needed.

Visual Notes

  • N/A

Content Usage

  • Offers should each have defined CTA’s. If an offer doesn’t have a defined CTA, it takes on the primary product stack CTA.

Pricing Substack

The pricing overview displays the final price and is only visible in a simplified verison of the product stack, typically for small or extra small screen sizes. Upon clicking more details, the full pricing sub stack is then displayed, and only then could the pricing sub stack and overview pricing be displayed together. The pricing sub stack and overview pricing can never be displayed together upon first arrival at the product stack.

Important! The pricing overview is automatically generated through the pricing sub stack. See the Pricing Sub Stack to learn about the legal requirements of the final price.

The pricing stack itself contains other subprocesses:

  • Base Price
  • Total Savings
  • Final Price
  • Currency Symbol

Base Price Subprocess

  • “Base Price” is a generic label that describes SAPP (Starting at Price Point).
  • MAPP stands for Minimum Advertised Price Point. This means that we cannot show the price to the customer until they have taken further action. That action is dependent on the manufacturer’s contracts with Dell.

Total Savings

  • Total Savings, Instant Savings, Aggregated Savings, and Instant Discounts used synonymously.
  • There are three types of savings that are counted in the Total Savings line: Instant Rebates, EPP (Employee Purchase Program), and Premier Specific Discounts.

Final Price Subprocess

  • “Final Price” is the price displayed after all instant discounts (inclusive of instant rebates, if applicable) have been applied. Remember that promo value is not subtracted from the Base Price value and thus not included in the Final Price value.
  • MAPP stands for Minimum Advertised Price Point. This means that we cannot show the price to the customer until they have taken further action. That action is dependent on the manufacturer’s contracts with Dell.

Shipping Substack

The asset/title/review is used to highlight a unique product attribute based on logic that incorporates legal requirements, lifecycle needs, timed events, and marketing differentiation.

Code/Logic

  • Optional. Allow ON only for items at SKU/OC level.
  • Automatically toggle OFF for items at Model and Franchise levels.
  • Text only. No special formatting.
  • Use an inheritance model to provide Bugs that can cascade across Franchise, Model, and OC levels.
  • In situations where there is no required legal need, display of the Bug is based on business logic and priority rankings. For example, business logic may specify that timed event Bugs can be overridden by ATS (available-to-sell) conditions, so that Cyber Monday Special text is replaced by Only [N] Left! when inventory levels hit a certain threshold.
  • Globalize as all other text per standard AppFramework globalization methods. Should read Left-to-Right or Right-to-Left based on language selection.

UX

  • Not clickable. No interaction.

Visual Notes

  • Plain text only, formatted per visual comps. No links; no interaction.
  • Text is centered or left-aligned to match with other elements in the stack.
  • Text wraps as necessary.

Content Usage

  • Business-controlled copy assigned at the OC/SKU level. Content is ideally concise to keep the bug on one line, without wrapping.
  • The stack label can be displayed in any Product Stack context, including search results, Browse, product carousels, and Cart.
  • When the stack label is manually maintained for each Order Code, the Business must create a solid content strategy for assigning useful differentiators to each OC. When all OCs on a page show the same bug text (for example, “Dell Recommended”), the stack label no longer serves its purpose as a useful differentiator.
  • DCX recommends that the stack label be suppressed in Product Stacks for items whose Product Titlesare also using differentiator terms (such as Good, Better, or Best or Fast, Ultra Fast, or Turbo Charged). This is to allow the user to focus on one key differentiator per item and avoid any potential conflicts between differentiator content in the Bug and the Title.
