General

Your code should not stick out — A consequent use of style, makes it easier for others to understand and use your HTML.

Indenting and white space

  • Set your editor to 4 spaces per tab.
  • Use tabs to indent blocks.
  • In general, you use one tab per tag level.
  • Avoid code lines longer than 80 characters.
  • Do not add blank lines without a reason.
  • For readability, add blank lines to separate large or logical code blocks.
  • Do not use unnecessary blank lines and indentation. It is not necessary to use blank lines between short and related items. It is not necessary to indent every element.

Use of Comments

  • Be generous with comments.
  • It is useful to leave information that will be read at a later time by people (possibly yourself) who will need to understand what you have done.
  • The comments should be well-written and clear, just like the code they are annotating.
  • It is important that comments be kept up-to-date.
  • Erroneous comments can make HTML code even harder to read and understand.
  • Make comments meaningful.
  • Short comments should be written on one line, with a space after <!-- and a space before -->
  • Long comments, spanning many lines, should be written with <!-- and --> on separate lines:
  • Long comments are easier to observe, if they are indented with a tab.

Use Lower Case Element Names

Although HTML5 allows mixing uppercase and lowercase letters in element names, All HTML elements should be written in lowercase for the following reasons:

  • Mixing uppercase and lowercase names is inconsistent and a bad practice.
  • Developers are used to using lowercase names (as in XHTML)
  • Lowercase looks cleaner and is easier to read.
  • Lowercase is easier to write
Good 

<section>
	...
</section>

Bad 

<Section>
	...
</SECTION>

Closing all HTML elements

  • Although HTML5 does not require closing some elements (such as the p tag) and some browsers are forgiving on other elements as well, ALL elements should be closed.
  • All empty HTML elements should be closed:
    • <input ... />
    • <hr/>
    • <br/>
    • <meta .../>

Use Lower Case Attribute Names

Although HTML5 allows mixing uppercase and lowercase letters in attribute names, All HTML attributes should be written in lowercase for the following reasons:

  • Mixing uppercase and lowercase names is inconsistent and a bad practice.
  • Developers are used to using lowercase names (as in XHTML)
  • Lowercase looks cleaner and is easier to read.
  • Lowercase is easier to write
Good

Custom Attribute Names should also be lower case and dasherized when incorporating multiple words.

<p class="foobar" custom-attribute="foobar">
	...
</p>

Bad 

<p class="foobar" CustomAttribute="foobar">
	...
</p>

Double Quote Attribute Names

Although HTML5 allows attribute values without quotes and with single or double quotes, you should double quote for the following reasons:

  • You have to use quotes if the value contains spaces
  • Mixing styles is never good
  • Quoted values are easier to read

  • Exeption to this rule:
    In cases where the attribute has HTML code content with double quotes, single quotes can be used to minimize having to escape double quotes inside the attribute value. Example: 

    
<a tabindex="1" role="popover" id="popover-right" data-trigger="focus" class="btn btn-default" data-html="true" data-toggle="popover" data-placement="right" data-content="Vivamus sagittis lacus vel augue laoreet rutrum faucibus." data-original-title="Popover on right <button type='button' class='close pull-right visible-phone' data-dismiss='popover'>×</button>">Popover on right</a>

Order of Attributes

HTML attributes should come in this particular order for easier reading of code.

  • class
  • id*, name
  • data-*
  • src, for, type, href, value
  • title, alt
  • role, aria-*

* IDs are very specific and should be used sparingly. In the demo code, IDs are used for identifying specific examples of code.

Image Attributes

  • Always use the alt attribute with images. It is important when the image cannot be viewed.
  • To make an image as wide as its container, use the image-responsive class available from dell-ui-bootstrap

Spaces and Equal Signs with Attributes

Although <span class = "foobar"></span> is perfectly legal a spaceless example <span class="foobar"></span> is much easier to read and groups attributes better together.

Filename Convention

  • All filenames should be lowercase (some systems like Unix are case sensitive)
  • All filenames should be dasherized (easier to read)
  • All filenames should have a .html extension (not .htm)
