Build and publish package

Publishing dell-ui-components is a multi step process. Some steps are automated and others are manual. Our effort is to eventually automate most of the steps but for now the following step by step instructions will help you publish a package.

About package versioning

Dell-ui-components uses Semantic Versioning to provide a simple set of rules for identifying changes in the package code. The changes fall into three different categories: MAJOR, MINOR and PATCH.

Step by step instructions

Please follow these instructions carefully.

Step 1 Open a git bash.

Open a git bash in your work directory for dell-ui-components. To do this, you can click on the working folder and click Git Bash

Step 2 Make sure you have the latest development version.

Follow the instructions below for confirming that everything is merged and in working order on development branch. If you need more help with this go to the git workflow section:

Checkout development branch

$ git checkout development

Pull the remote code from origin and merge with your local branch.

$ git pull origin development

If a conflict occurs then fix and start over.

Test merged code

$ grunt serve

You should be able to view demo at

Step 3 Merge everything to staging

Now we will switch to the staging branch and merge development code into it.

Checkout staging branch

$ git checkout staging

Pull the remote code from origin and merge with your local branch.

$ git pull origin staging

If a conflict occurs then fix and restart step.

Merge development code into staging

$ git merge development

If needed, resolve merge issues then commit

Test merged code

$ grunt serve

You should be able to view demo at

Step 4 Revision and release notes

Version number and release notes are very important in the publishing process so read carefully:

Edit the package.json file with new version number and follow guidance above for semantic versioning.

Edit README.md file with new release notes

Save all changes locally

$ git add –A && git commit –m"Added release notes for version 1.0.x"

Save all changes to remote server

$ git push origin staging

Step 5 Build code

The build command brings everything together.

$ grunt build

NOTE It might fail because it can’t delete certain directories like dist. Delete manually the dist directory then grunt build again

The build command brings everything together.

$ grunt build

Step 6 Test build

Now we start the server and check the built demo.

$ grunt serve

You should be able to view built demo at

NOTE Repeat step 5 and 6 if changes are necessary

Step 7 Test build on staging.delldesignlibrary.com

staging.delldesignlibrary.com automatically uses package files that are hosted on ui.sherpaux.com. Update these files via FTP and the pages on staging and then review components on staging.delldesignlibrary.com to verify everything is in order.

Step 8 Save staged code

The build command brings everything together.

Save all changes locally

$ git add –A && git commit –m"Final build for version 1.0.x"

git push origin staging

Step 9 Merge final build with development and final check

Now we merge the build back to development to assure all future development is in synch.

$ git checkout development

$ git merge staging

$ grunt serve

NOTE If anything is wrong (unlikely) then you need to start over

Step 10 Merge final build with master for final publishing

Now we merge the complete final tested build from staging to master

$ git checkout master

$ git pull origin master

$ git merge –X theirs staging (if for any reason you have conflicts resolve with "theirs")

$ git push origin master

NOTE If anything is wrong (unlikely) then you need to start over

Step 11 Update package branch

The best way to approach this step is to have a different folder that has the package branch checked out.

Step 12 Save changes to package branch

Once files are updated in the package branch we can commit, tag and push.

$ git add –A && git commit –m"Package version 1.0.x"

$ git push origin package

$ git tag 1.0.x

$ git push origin 1.0.x

Step 13 Get outside of the corporate firewall.

Git interactions with github use a particular transfer protocol and port to download files that is typically blocked by corporate firewall. The next few steps require this so you must have access outside of the corporate firewall.

NOTE If you are logged in through VPN you can skip this.

Step 14 Push code to Github

By pushing the code to github it effectively publishes the package to Bower.

$ git tag 1.0.x

$ git push github 1.0.x

Step 15 Confirm Bower package is published

You might have to wait a minute or so to check this.

$ bower info dell-ui-components

In the result list of bower packages you should see the new version

Step 16 Go back inside the corporate firewall

NOTE If you are logged in through VPN you can skip this.

Step 17 Final step

You are done!

You can check to make sure the package is saved in our internal server. Go to: https://github.com/DellGDC/dell-ui-components/tree/1.0.x (1.0.x being whatever version you just published)
